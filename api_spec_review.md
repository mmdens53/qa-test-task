# Review спецификации микросервиса продажи билетов

## Найденные проблемы спецификации

1. Не указан формат данных запросов и ответов (JSON, XML и т.д.).  
   Влияние: невозможно однозначно определить формат тела запросов и формировать корректные тестовые данные.

2. Не описаны параметры пути и тела запроса.  
   Пример: не определено, является ли `{eventId}` числом или строкой; не перечислены обязательные поля для POST-запроса (id события, пользователь, количество билетов и т.п.).  
   Влияние: затруднена проверка валидации и граничных значений.

3. Отсутствуют примеры успешных и неуспешных ответов.  
   Влияние: нельзя заранее определить структуру `response`, возможные поля и их типы.

4. Не описаны требования к авторизации и аутентификации.  
   Влияние: неясно, могут ли неавторизованные пользователи совершать покупку и какие коды ошибок будут возвращаться при отсутствии прав.

5. Не указаны обязательные и необязательные поля в запросах.  
   Влияние: невозможно проверить корректную обработку неполных запросов и сообщений об ошибках валидации.

6. Не прояснена логика покупки нескольких билетов за один запрос и поведение при отсутствии доступных мест.  
   Влияние: невозможно протестировать сценарии перегрузки, частичной покупки и корректные коды ошибок при нехватке билетов.

## Рекомендации по доработке спецификации

- Явно указать формат данных (например, `application/json` для всех методов).  
- Описать схемы запросов и ответов (поля, типы, обязательность).  
- Добавить примеры успешных и ошибочных ответов с реальными кодами статусов HTTP.  
- Описать требования к авторизации (тип токена, роли, возможные ошибки авторизации/аутентификации).  
- Уточнить бизнес-логику покупки нескольких билетов и сценарии при исчерпании квоты.
