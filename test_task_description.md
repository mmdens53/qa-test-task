# Описание тестового задания

Документ содержит формулировку исходного тестового задания для стажера QA (ручное тестирование).  
Цель заданий — оценить подход к решению задач, аналитическое мышление, внимательность, умение рассуждать, искать информацию и формулировать выводы.

Результаты всех заданий предполагается оформлять в виде документа с ответами и пояснениями к каждому шагу.

## Задание 1. Тест-анализ и декомпозиция

**Описание функциональности интернет-магазина**

> Пользователь добавляет товар в корзину.  
> В корзине он может изменить количество этого товара, а также удалить его.

**Что нужно сделать**

1. Составить чек-лист для проверки указанной функциональности.  
   Чек-лист должен описывать возможные сценарии проверки и быть понятным для другого тестировщика.
2. Разделить сценарии на группы по выбранному принципу  
   (например, позитивные/негативные проверки или по элементам интерфейса).  
   Описать, почему выбран именно такой принцип группировки.

## Задание 2. Баг-репорт и исследование

**Исходные данные**

Есть форма регистрации нового пользователя со следующими элементами:

- Поле «Имя»  
- Поле «E-mail»  
- Поле «Пароль»  
- Поле «Подтверждение пароля»  
- Кнопка «Зарегистрироваться»

Точные требования к работе формы не заданы — необходимо продумать проверки самостоятельно.

**Что нужно сделать**

1. Описать идеи для проверки формы регистрации: какие сценарии стоит попробовать.  
2. Выбрать три проверки, которые будут выполнены в первую очередь, и объяснить выбор.  
3. Рассмотреть ситуацию, когда после нажатия на кнопку «Зарегистрироваться» ничего не происходит,  
   и описать свои дальнейшие шаги: что будет проверено и какие вопросы будут заданы.  
4. Заполнить баг-репорт по следующей структуре (в симуляционных полях указать «не применимо» или «неизвестно»,  
   а там, где возможно, сформулировать гипотезы):

   - **Заголовок:** краткое и понятное описание проблемы  
   - **Шаги воспроизведения:** пронумерованный список действий  
   - **Фактический результат:** что произошло на самом деле  
   - **Ожидаемый результат:** что должно было произойти  
   - **Критичность:** высокая / средняя / низкая, с обоснованием выбора  
   - **Дополнительная информация:** дополнительные наблюдения, возможные причины ошибки,  
     а также данные, которые могли бы помочь при анализе (например, информация из инструментов разработчика)

## Задание 3. Техническое расследование

**Исходный псевдокод**

```js
function checkAccess(age, hasTicket, invited) {
  if (age < 18) {
    return "Доступ запрещен";
  }

  if (hasTicket == false && invited == false) {
    return "Нет билета или приглашения";
  }

  if (hasTicket == true || invited == true) {
    return "Доступ разрешен";
  }
}
```

## Задание 4. SQL-запросы

```sql
-- 1. Все товары, цена которых больше 10 000 рублей
SELECT *
FROM products
WHERE price > 10000;

-- 2. Список всех товаров вместе с названием категории
SELECT 
    p.id,
    p.name           AS product_name,
    p.price,
    c.name           AS category_name
FROM products p
LEFT JOIN categories c ON p.category_id = c.id;
-- 3. Количество товаров в каждой категории
SELECT 
    c.name                  AS category_name,
    COUNT(p.id)             AS products_count
FROM categories c
LEFT JOIN products p ON c.id = p.category_id
GROUP BY c.id, c.name
ORDER BY products_count DESC;
-- 4. Товары, отсортированные по цене от дешёвых к дорогим — только первые три
SELECT *
FROM products
ORDER BY price ASC
LIMIT 3;
```

## Задание 5. Рецензия спецификации микросервиса «Pokupka Bileta»

### Основные найденные проблемы и рекомендации

1. **Отсутствует информация об аутентификации и авторизации**  
   - Кто может вызывать методы?  
   - Требуется ли JWT, API-ключ, OAuth, сессия?  
   - Возможен ли доступ для неавторизованных пользователей?

2. **Метод GET /api/events/{eventId}**  
   - Не указан тип и формат `eventId` (integer, UUID, string и т.д.)  
   - Нет примера полного URL  
   - Отсутствует пример ответа 200 OK (структура JSON, обязательные поля)  
   - Не описаны возвращаемые поля события (название, дата, время, цена, количество мест и т.п.)

3. **Метод POST /api/tickets**  
   - Полностью отсутствует описание структуры тела запроса (body schema)  
   - Нет ни одного примера запроса (JSON-body)  
   - Неясно, какие поля обязательны, какие опциональны  
   - Возможные поля (гипотезы): `eventId`, `quantity`, `pricePerTicket`, `userId`, `paymentMethod` и др.

4. **Недостаточно детализированы коды ответов**  
   - 201 Created — должен возвращаться ли `Location` header? ID созданного билета?  
   - 400 Bad Request — какие именно сообщения об ошибках? В каком формате?  
   - 404 Not Found — что именно не найдено (мероприятие, билет, пользователь)?  
   - Отсутствуют важные коды: 401, 403, 409 (конфликт — билеты закончились), 429, 503

5. **Нет примеров запросов и ответов**  
   - Ни одного curl / HTTP-запроса с телом  
   - Ни одного примера успешного и ошибочного ответа

6. **Отсутствует версионирование API**  
   Рекомендуется использовать хотя бы `/api/v1/...`

7. **Нет упоминания идемпотентности**  
   - При повторной отправке одного и того же запроса на покупку — что должно происходить?  
   - Рекомендуется поле `idempotencyKey`

8. **Нет бизнес-ограничений и правил**  
   - Максимальное количество билетов в одном заказе?  
   - Можно ли купить билет на уже начавшееся / закончившееся мероприятие?  
   - Что происходит при ошибке оплаты / недостатке мест?

9. **Нет информации о формате данных**  
   - Формат дат (ISO 8601?)  
   - Валюта (RUB, EUR?) и её отображение  
   - Локализация (язык ответов)

**Вывод и рекомендация**  
Спецификация в текущем виде **непригодна** для полноценной реализации и тестирования.  
Для продолжения работы необходимо дополнить документ как минимум:

- OpenAPI/Swagger спецификацией или JSON Schema  
- Полными примерами запросов и ответов (включая ошибки)  
- Описанием авторизации  
- Бизнес-правилами и ограничениями  
- Списком всех возможных ошибок с кодами и сообщениями

Без этих доработок разработка и тестирование будут проходить в режиме «угадайки».
