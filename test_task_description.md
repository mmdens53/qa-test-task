# Описание тестового задания

Документ содержит формулировку исходного тестового задания для стажера QA (ручное тестирование).  
Цель заданий — оценить подход к решению задач, аналитическое мышление, внимательность, умение рассуждать, искать информацию и формулировать выводы.

Результаты всех заданий предполагается оформлять в виде документа с ответами и пояснениями к каждому шагу.

---

## Задание 1. Тест-анализ и декомпозиция

**Описание функциональности интернет-магазина**

> Пользователь добавляет товар в корзину.  
> В корзине он может изменить количество этого товара, а также удалить его.

**Что нужно сделать**

1. Составить чек-лист для проверки указанной функциональности.  
   Чек-лист должен описывать возможные сценарии проверки и быть понятным для другого тестировщика.
2. Разделить сценарии на группы по выбранному принципу  
   (например, позитивные/негативные проверки или по элементам интерфейса).  
   Описать, почему выбран именно такой принцип группировки.

## Задание 2. Баг-репорт и исследование

**Исходные данные**

Есть форма регистрации нового пользователя со следующими элементами:

- Поле «Имя»  
- Поле «E-mail»  
- Поле «Пароль»  
- Поле «Подтверждение пароля»  
- Кнопка «Зарегистрироваться»

Точные требования к работе формы не заданы — необходимо продумать проверки самостоятельно.

**Что нужно сделать**

1. Описать идеи для проверки формы регистрации: какие сценарии стоит попробовать.  
2. Выбрать три проверки, которые будут выполнены в первую очередь, и объяснить выбор.  
3. Рассмотреть ситуацию, когда после нажатия на кнопку «Зарегистрироваться» ничего не происходит,  
   и описать свои дальнейшие шаги: что будет проверено и какие вопросы будут заданы.  
4. Заполнить баг-репорт по следующей структуре (в симуляционных полях указать «не применимо» или «неизвестно»,  
   а там, где возможно, сформулировать гипотезы):

   - **Заголовок:** краткое и понятное описание проблемы  
   - **Шаги воспроизведения:** пронумерованный список действий  
   - **Фактический результат:** что произошло на самом деле  
   - **Ожидаемый результат:** что должно было произойти  
   - **Критичность:** высокая / средняя / низкая, с обоснованием выбора  
   - **Дополнительная информация:** дополнительные наблюдения, возможные причины ошибки,  
     а также данные, которые могли бы помочь при анализе (например, информация из инструментов разработчика)

## Задание 3. Техническое расследование

**Исходный псевдокод**

```js
function checkAccess(age, hasTicket, invited) {
  if (age < 18) {
    return "Доступ запрещен";
  }

  if (hasTicket == false && invited == false) {
    return "Нет билета или приглашения";
  }

  if (hasTicket == true || invited == true) {
    return "Доступ разрешен";
  }
}
```

**Вопросы:**
1. Что делает эта функция? Опиши своими словами, какую задачу она решает.
2. Найди логическую ошибку (если она есть) и объясни, почему это может привести к
неверному результату.
3. Как бы ты улучшил этот код, чтобы он работал надежнее или понятнее (словами, без
кода)?

## Задание 5. Проверка спецификации микросервиса

**Контекст**

На проверку передано описание (спецификация) нового микросервиса, который должен быть реализован в реальной системе.  
Аналитик просит предварительно изучить спецификацию и сообщить обо всех найденных ошибках и неясностях, чтобы скорректировать документ до разработки.  

**Что нужно сделать**

1. Найти все ошибки, несоответствия и странные моменты в примере спецификации.  
2. Составить их список с развернутыми комментариями, чтобы аналитик точно понял, что необходимо исправить или уточнить.  

### Спецификация API для микросервиса «Pokupka Bileta»

**Общая информация**

Микросервис «Pokupka Bileta» предоставляет функционал для покупки билетов на мероприятия.  
API включает два метода:

- получение информации о мероприятии;  
- покупка билета.  


#### 1. Метод: Получение информации о мероприятии

- **URL:** `/api/events/{eventId}`  
- **Метод:** `GET`  
- **Описание:** получает информацию о мероприятии по его идентификатору.  

**Параметры запроса**

- `eventId` — идентификатор мероприятия (формат явно не определён).

#### Примеры запросов

```http
GET /api/events/12345
```

#### Примеры ответов

**Успешный ответ (200 OK)**
```
{
  "id": "12345",
  "name": "Концерт",
  "date": "2025-10-20T19:00:00Z",
  "location": "Москва"
}
```

**Мероприятие не найдено (404 Not Found)**
```
{
  "error": "Event not found"
}
```

**Возможные статусы и ошибки**

- `200` OK — информация о мероприятии успешно получена.
- `400` Bad Request — неверный формат идентификатора мероприятия.
- `404` Not Found — мероприятие с указанным идентификатором не найдено.
- `500` Internal Server Error — внутренняя ошибка сервера.

#### 2. Метод: Покупка билета

- **URL:** /api/tickets
- **Метод:** POST

Описание: позволяет приобрести билет на мероприятие.

### Body запроса

**Тело запроса передаётся в формате application/json.**

```json
{
  "eventId": "12345",
  "userId": "67890",
  "quantity": 2
}
```
#### Примеры запросов

```http
POST /api/tickets
Content-Type: application/json

{
  "eventId": "12345",
  "userId": "67890",
  "quantity": 2
}
```

#### Примеры ответов

**Успешный ответ (201 Created)**
```json
{
  "message": "Ticket purchased successfully"
}
```
**Некорректный запрос (400 Bad Request)**
```json
{
  "error": "Invalid request body"
}
```
**Мероприятие не найдено (404 Not Found)**
```json
{
  "error": "Event not found"
}
```

### Возможные статусы и ошибки

- `201` Created — билет успешно куплен.
- `400` Bad Request — ошибка в теле запроса или его структуре.
- `404` Not Found — указанный ресурс (например, мероприятие) не найден.
- `500` Internal Server Error — внутренняя ошибка сервера при обработке запроса.
